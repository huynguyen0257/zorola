Update after inspecting copied macOS debug artifacts:

- Debug folder inspected: `data/debug/2026-04-15T18-12-05-241Z/`.
- Zalo proof images are present as normal `<img>` elements with:
  - `className = "zimg-el"`
  - `src = "blob:https://chat.zalo.me/..."`
  - rendered dimensions such as `148x319`, `146x319`, and `110x238`.
- The previous candidate filter required `width >= 180`, so it rejected narrow portrait phone screenshots even though they were real proof images.
- Candidate filtering was changed to accept images by area and side lengths:
  - area at least `20_000`
  - longest side at least `180`
  - shortest side at least `100`
- Simulating the new filter against `report.json` selects 5 likely proof images from that debug capture.
- After the user synced the Mac `data` folder back to the server, 3 collected JPEG files were present under `data/data/images/collected/`.
- Those 3 files are valid original images around `592x1280`. Two contain visible account numbers, and one is an account-open success screen.
- To explain future differences between candidate count and saved file count, collection now writes a run report at `data/runs/<timestamp>/collect-report.json`.
- The report records every candidate, every saved hash, and whether an image was skipped as already known.
- A later screenshot showed the 20:35 cluster above the 22:09 cluster. The collector previously scrolled up all rounds before collecting, which is unsafe with Zalo's virtualized message list because newer visible messages can leave the DOM after scrolling.
- The collector flow was changed to collect images from the current viewport first, then scroll one round and collect again after each scroll round.
- A later macOS run collected many images across several scroll positions, but crashed on round 5 when one candidate disappeared before fallback screenshot capture:
  - `locator.screenshot: Timeout 5000ms exceeded`
  - candidate example: `[data-zorola-candidate-index="img-60"]`
- This is expected with Zalo's virtualized DOM: a candidate can be discovered and then unmounted before it is captured.
- Candidate capture was changed so one failed image is recorded in the run report and logged as a warning instead of crashing the whole batch.
- The next refactor removed the stale-locator capture path for the main flow. The collector now tries to capture bytes immediately inside the browser context when candidates are discovered, with a canvas fallback for mounted `<img>` elements.
- Candidate metadata now includes `messageTime` when it can be inferred from nearby DOM text, so any remaining failed candidate can be searched manually in Zalo by time.
- Collection is also capped to `30` images per scan step to keep test runs faster.
